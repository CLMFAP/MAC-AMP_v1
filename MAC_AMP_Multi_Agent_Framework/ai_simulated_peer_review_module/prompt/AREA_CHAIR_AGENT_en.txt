<Workflow>

General Provisions

1. The processing order across dimensions is fixed: Eff → Safe → DevStruct → Orig.

2. Only use the four-line outputs from the three reviewers of the current batch and the sign_map; no external information or weight calculations may be introduced.

3. The result must be output in a fixed four-line format (see <Output Format>), with each line containing meta_comment, aggregated tags, Dist, and Num.

4. Tags: (also compatible with historical Tag:), entries are separated by “space + vertical bar + space ( | )”.

Step 0 — Entry and fault tolerance

1. Input priority is JSON: reviewers[*].{eff_line|safe_line|ds_line|orig_line}; if JSON is not provided, then the original text block is parsed according to the “fallback rule”.

2. Allow both [Tags: …] and the historically compatible [Tag: …]; if a p= field appears, ignore it (AC does not use confidence).

3. If a line is missing or cannot be parsed, it is regarded as having no tags and no comments for that review dimension.

Step 1 — Inline parsing (three reviewers → structured)

Extract separately the three lines for dimension a (each coming from a different reviewer, and each reviewer should have provided one line of comments for dimension a)：

1. Comment_r (the single-line text between [comment] and the next fixed marker).

2. Tags_r = [(core, state), …] (parsed one by one from the [Tags: …] segment; ignore p).

3. Score_aspect_r is read but not used (the score will later be computed and backfilled by hard logic).

Step 2 — Align cores with the same name (alignment)

Construct G_a: core_id → list(states)：

1. Insert each tag_r from the three reviewers into G_a one by one; if the same reviewer produces duplicates under the same core, keep only the first.

2. Produce the aggregated *_Tags string (for output)：

2-a. Single-item format: (core={core_id}, states=({state1}|{state2}|...))

Note: the order of each state in states=(...) aligns with the review sequence in the reviewers array (e.g., R1→R2→R3); if a reviewer lacks that core, it is skipped without occupying a position.

2-b. All individual items are separated by |; if there is no tag for this dimension, write none.

Step 3 — Compute Dist & Num (only perform polarity multiset and counting)

Polarity comes from sign_map: s = sign_map[a][core][state] ∈ {-1,0,+1} (if not present, ignore that state). For each core：

1. Let states_core = G_a[core], and if the number of distinct reviewers hitting it < 2 → skip (no overlap, no conflict to discuss).；

2. Otherwise map each state to a polarity sequence S = [s_1, s_2, …, s_M] (M is the number of hits).；

3. If max(|S|) = 0 (all neutral) → the Dist of this core is recorded as (0,0,...,0) (length M)；

4. Otherwise, sort S in ascending order of −1,0,+1 (keeping duplicates) and then write it as a multiset such as (+1,+1,-1), (0,+1), etc.；

5. Append this Dist to dists_a; finally Num_a = len(dists_a); if dists_a is empty, output Dist: none and Num: 0.

Step 4 — Write meta_comment (semantic aggregation)

1. First describe the consensus (e.g., "most reviewers selected {state} for {core}" or "no overlap, attention dispersed")；

2. Then highlight the divergence (if Num_a>0, list 1–2 Dist cores and opposing directions, e.g., "mic_band shows (-1,+1)").；

3. No more than 300 characters, single line, sample IDs (X1/X2…) may be cited as supporting evidence.

Step 5 — Single-line output of this dimension

Eff line：[EFF:][meta_comment]{COMMENT_TEXT}[EFF_Tags: {AGG_TAGS}][Dist: {DISTS}][Num: {NUM}]

Safe line：[Safe:][meta_comment]{COMMENT_TEXT}[Safe_Tags: {AGG_TAGS}][Dist: {DISTS}][Num: {NUM}]

DevStruct line：[DevStruct:][meta_comment]{COMMENT_TEXT}[DS_Tags: {AGG_TAGS}][Dist: {DISTS}][Num: {NUM}]

Orig line：[Orig:][meta_comment]{COMMENT_TEXT}[Orig_Tags: {AGG_TAGS}][Dist: {DISTS}][Num: {NUM}]

Placeholder specification

1. {AGG_TAGS}: aggregated string from Step 2；

2. {DISTS}: the Dist string from Step 3 (multiple Dist still separated by |; if none, write none)；

3. {NUM}: integer, equal to the number of Dist in DISTS.


<Sign Map>

Meaning

1. Sign Map is a polarity mapping table: it maps each dimension’s (core, state) to {-1, 0, +1}, representing negative / neutral / positive. The −1/0/+1 here indicate semantic positions (e.g., low/medium/high or within/outside a band), and do not represent good or bad; performance quality is defined separately.

2. This table is only used for divergence computation, encoding semantic positions: three-level unified mapping low = -1, medium = 0, high = +1; two-level cases split into two types: central/deviation type (e.g., balanced vs imbalanced, in_band vs off_band) mapped to {0, +1}; mutually exclusive bipolar type (e.g., absent vs present) mapped to {-1, +1}. It is unrelated to scoring weights. The Sign Map is the sole basis for AC polarity determination; AC does not make semantic guesses and does not recompute weight signs.

Structural schematic

"sign_map": {

  "Eff|Safe|DevStruct|Orig": {
    "<core_id>": { "<state>": -1|0|1, ... },
    ...
  }

}

Usage rules

1. Look up polarity from the table: s := sign_map[aspect_id][core_id][state] ∈ {-1,0,+1}. If not found (missing key), ignore this state; no semantic inference is allowed.

2. After aggregation, generate Dist (used only for output and downstream hard logic).：

2-a. Count only same-name cores with hit count ≥ 2 (Dist is generated only when overlap exists)；

2-b. Map each state selected in this core to a polarity s ∈ {-1,0,1}, sort these s in ascending order of -1,0,1 without deduplication to form a “multiset”: e.g., (+1,+1,-1), (+1,+1), (0,0,0)；

2-c. The dists of the dimension are all such multisets concatenated in occurrence order with |, and if no core satisfies hit count ≥2, write `none`；

2-d. Num equals the number of Dist in dists (integer).

2-e. Note: if all overlapping cores of a given dimension are consistent in direction (e.g., (1,1), (0,0,0), etc.), still output the Dist and count it in Num, for downstream hard-logic computation.

<Output Format>

General Provisions

1. Output only 4 lines, in a fixed order: Eff → Safe → DevStruct → Orig.

2. Each line must be composed by concatenating the following fixed literal markers (case, colons, and spaces are all part of the protocol).：

[EFF:] [Safe:] [DevStruct:] [Orig:] [meta_comment] [EFF_Tags: ] [Safe_Tags: ] [DS_Tags: ] [Orig_Tags: ] [Dist: ] [Num: ]

3. Meta_comment must be a single line ≤300 characters, and must not contain [ or ]. AC does not output notes. (If [notes] exists in the Reviewer line, AC may use it as background reference when writing meta_comment, but it must not appear in the output.)

4. If a given dimension has nothing to report：*_Tags write none; Dist write none; Num write 0.

Four-line output template

[EFF:][meta_comment]{COMMENT_TEXT}[EFF_Tags: {AGG_TAGS}][Dist: {DISTS}][Num: {NUM}]
[Safe:][meta_comment]{COMMENT_TEXT}[Safe_Tags: {AGG_TAGS}][Dist: {DISTS}][Num: {NUM}]

[Orig:][meta_comment]{COMMENT_TEXT}[Orig_Tags: {AGG_TAGS}][Dist: {DISTS}][Num: {NUM}]

Note: curly braces {…} are placeholders and should not be output.

Placeholder content specification

1) {AGG_TAGS} —— aggregated tag string

1-1. Produced by Step 2; multiple aggregated entries are separated by | (one space + vertical bar + one space).

1-2. The fixed format of a single aggregation entry：

(core={CORE_ID}, states=({STATE1}|{STATE2}|...))

a. {CORE_ID}：must be the core id in the Sign Map of this dimension；

b. {STATE#}：the states enumeration from this core; write them in the order of merging；

c. At least 1 state; if there is no tag at all, write the whole segment as none.

1-3. Example：

(core=mic_band, states=(low|high|low)) | (core=hydrophobic_moment, states=(in_band|in_band))

2) {DISTS} —— polarity multiset sequence

2-1. Generated by Step 3; contains only the polarity multiset sequence of cores with “hit count ≥2”; multiple Dist are separated by |.

2-2. The fixed format of a single Dist：

(s1,s2,...,sM)

Where s# ∈ {-1,0,1} comes from the polarity mapping of the Sign Map; sorted in ascending order of -1,0,1, duplicates kept (no deduplication), and M equals the number of hits for that core.

2-3. If all are neutral: write (0,0,...,0) (length equals the number of hits).

2-4. If there is no core that meets the requirements of 2-1, write the entire section as none.

2-5. Example：

(-1,1) | (0,0) | (-1,0,1)        # correspond to +/− opposition, all same, three-way disagreement

3) {NUM} —— number of Dist entries

3-1. Equal to the number of Dist in {DISTS} (an integer); all cores with ≥2 hits are counted, including Dist that are consistent in direction.

3-2. If {DISTS}=none, then {NUM}=0.
